## Repair native-arch breakage, don't re-`pnpm remove` ‚Äî one remove can swap lightningcss to the wrong arch and kill the CSS build

- **Context**: Any dependency op (`pnpm remove <pkg>`, casual `pnpm install`) on this Apple-Silicon (darwin arm64) machine ‚Äî especially the "remove unused deps" cleanup steps from the simplify audit. `lightningcss` is a transitive native dep of `@tailwindcss/postcss` (Tailwind v4) with one optional prebuilt package per platform.
- **Problem**: `pnpm remove recharts` re-linked `node_modules` and pulled `lightningcss-darwin-x64` (Intel) instead of `lightningcss-darwin-arm64`, dropping `lightningcss.darwin-arm64.node`. `@tailwindcss/postcss ‚Üí lightningcss` then threw `Cannot find module '../lightningcss.darwin-arm64.node'` and Turbopack failed to evaluate `src/styles/globals.css` ‚Äî the whole dev CSS build broke. Everything reports arm64 (node, pnpm's runtime, the binary itself), the lockfile lists every platform optional incl. arm64, and there's no arch config ‚Äî yet a plain `pnpm install` no-ops (thinks `node_modules` is in sync) and leaves the stale x64 artifact. ~15‚Äì20 min to diagnose because (a) the error blames `globals.css` (looks like a CSS/source bug) when the cause is install-state, and (b) the fix has TWO parts and the first looks complete on its own. Tell in the `pnpm remove` output: `reused 0, downloaded 948` = it rebuilt the store from scratch.
- **Rule**: Prefer hand-editing `package.json` deps over `pnpm remove`/`pnpm install` on this repo. When a dependency op is unavoidable, expect native-arch breakage and repair in **both** layers: (1) **binary** ‚Äî `pnpm install --force` (re-links optionals for the real arch; ~3‚Äì4 min), confirm `node_modules/.pnpm/lightningcss-darwin-arm64@*/node_modules/lightningcss-darwin-arm64/lightningcss.darwin-arm64.node` exists; (2) **cache** ‚Äî the running `next dev` keeps failing from `.next/dev` even after the binary is fixed, so `rm -rf .next` and restart the dev server. Critically: a bare `require('lightningcss')` or even a standalone `@tailwindcss/postcss` compile can succeed while the dev server still errors ‚Äî **verify against the actual running dev server, not a node one-liner** (same family as the verify-against-reality / don't-trust-a-stale-harness rule). When an error blames a CSS/source file right after a dependency op, suspect install-state before re-diagnosing the code.
- **Applies to**: /10x-implement, /10x-impl-review, /simplify, dead-code-scanner, any dep-removal cleanup on this machine

## Treat kosztorys sheet column positions as a frozen external contract

- **Context**: Any change to the Google Sheets sync algorithm (`src/lib/google/sheets.ts`, `src/lib/actions/sheets-sync.ts`, `src/hooks/transfers/sync-sheet.ts`) that affects what gets written to a tab ‚Äî especially the summary/SUMIF block layout, column order, or the lists that drive it (`SHEET_TRANSFER_TAB_TYPES`, `TRANSFERS_SUMMARY_TYPES`, `transferSummaryKeys`, `*_TAB_CONFIG.header`). Holds until the sheets are removed entirely.
- **Problem**: Kosztorys sheets across many client spreadsheets contain hand-built formulas that reference summary columns by fixed position. Dropping or reordering a column shifts the survivors, silently breaking those formulas ‚Äî and there is no fix except going sheet-by-sheet by hand. This already happened: removing CORRECTION from `SHEET_TRANSFER_TAB_TYPES` (a data-routing list) also fed `transferSummaryKeys()`, so a tab rebuild would have shrunk the summary and slid Strata into Korekta's slot.
- **Rule**: Never let a column leave or move in a written sheet layout. When data routing must change, keep the column in place (a zero-total placeholder label is fine) and decouple the layout list from the routing list. Audit every consumer of the changed constant before editing it ‚Äî one list often drives both routing and layout. Guard the frozen layout with a test that asserts exact column order/count.
- **Applies to**: plan, plan-review, implement, impl-review

## A parity test must run the REAL per-surface assembly on REAL data ‚Äî a shared stand-in is false confidence

- **Context**: Any figure shown in more than one place computed by more than one code path ‚Äî here the investor `bilans`/`mar≈ºa` on the investments listing (`lib/queries/investments.ts` ‚Üí `calculateBalance`/`calculateMargin`) vs the detail/report page (`financial-stats` ‚Üí `ToggleStatButtons` interactive sum + `calculateMargin`) vs print/export (`lib/export/header-fields.ts`). Two functions are even both named `calculateBalance` (a static formula vs a sum-of-visible-cards). Applies to any future change touching these figures.
- **Problem**: Equality between independently-assembled copies of the same figure is an unenforced coincidence that decays on every change. We shipped a "parity test" that compared `extractFigures ‚Üî extractFigures` ‚Äî a helper **no page uses** ‚Äî so it stayed green while the listing and detail pages disagreed by hundreds of z≈Ç on 7 real investments (legacy un-categorised corrections: counted by the listing's `totalMaterialCosts`, absent from the detail's category-card sum). Repeatedly declared "verified / can't break" from reasoning about the wrong layer; only real/legacy data exposed the gap. Also mis-stated the duplication ("4‚Äì6 places") from memory instead of grep (truth: mar≈ºa = 1 formula, 3 call sites incl. an orphaned `extractFigures`; bilans = 2 legitimate algorithms ‚Äî the detail's interactive toggle vs the listing's static formula).
- **Rule**: (1) A parity/regression test must call the **actual functions the surface renders**, on **real data** (gated DB test), not a stand-in. (2) Prove it **red** ‚Äî it must fail when you break the real thing; a test that's only ever green may be tautological. (3) A _legitimate_ difference (interactive toggle vs static formula) is fine ‚Äî but then assert the invariant (default/all-visible == formula) with a test; don't assume. (4) Never claim "verified" without an executable red‚Üígreen on the real path. (5) Count call sites with `grep`, never from memory. (6) Consolidate duplicated assembly into one function that takes the whole `financials` object (`calculateBalance(financials)`, `calculateMargin(financials)`) ‚Äî no long positional-arg lists, no vaguely-named orphan bundles.
- **Applies to**: plan, plan-review, implement, impl-review, tdd, simplify

## react-datasheet-grid in a flex container flickers ‚Äî give it a definite width with `grid-cols-[minmax(0,1fr)]`

- **Context**: Mounting `react-datasheet-grid` (`DataSheetGrid`/`DynamicDataSheetGrid`) full-height inside a flexbox layout (`flex flex-col` + `flex-1`), e.g. the kosztorys v2 editor (`src/components/kosztorys/kosztorys-editor-v2.tsx`). Any datasheet-grid where the sum of column min-widths exceeds the viewport.
- **Problem**: The whole page flickered ("blinking navbar / Kolumny button"), DevTools Issues climbed ~900/sec (silent ResizeObserver-loop issues, no console error). Cause: the grid's column min-widths (~1650px) exceed the viewport (~1200px); in a flex context the container width has no _definite_ size, so it oscillated viewport‚Üîcontent (measured 1200‚Üî1653 **every frame**) and the grid's internal `react-resize-detector` chased it forever. ~30 min lost theorizing about ResizeObserver/`router.refresh` loops; the truth only showed up by sampling `getBoundingClientRect().width` over a 2s rAF loop and seeing two alternating values. A `min-w-0` + `overflow-hidden` on the wrapper did **not** fix it.
- **Rule**: Give the grid's container a **definite** width so its resize-detector settles: wrap it in a CSS-grid track `className="grid grid-cols-[minmax(0,1fr)] ‚Ä¶"` (the `minmax(0,1fr)` ‚Äî not the default `1fr`=`minmax(auto,1fr)` ‚Äî forbids expand-to-content; the grid then scrolls its columns internally). Diagnose width/height loops by sampling `getBoundingClientRect()` across `requestAnimationFrame` for 1‚Äì2s and checking for >1 distinct value ‚Äî don't rely on console errors (Chrome reports RO loops only in the Issues panel). Separately: feed the grid a `height` decoupled from its own box (`window.innerHeight ‚àí rect.top`, measured on mount + window-resize, **no** ResizeObserver), and only call `router.refresh()` in `onChange` on a real edit.
- **Applies to**: implement, impl-review, verify, frontend grid/table work

## react-datasheet-grid has no native column resize and ignores runtime sizing-prop changes ‚Äî remount it with a `key`

- **Context**: Any user-resizable / programmatically-resized column on `react-datasheet-grid` (4.x), e.g. the kosztorys v2 editor's drag-to-resize (`src/lib/tables/kosztorys-v2-columns.tsx`, `src/components/kosztorys/{column-resize-handle,use-column-widths}.tsx`). The library has **no** `resizable` prop or drag handle ‚Äî only flex sizing (`basis`/`grow`/`shrink`/`minWidth`/`maxWidth`).
- **Problem**: Changing a mounted column's sizing props at runtime (e.g. pinning `basis`/`minWidth`/`maxWidth` after a drag) does **not** re-lay-out the grid, even though `useColumnWidths` memoizes on a content-hash of `[basis,minWidth,maxWidth,grow,shrink]` and we pass a fresh `columns` array each render. The new width reaches the columns prop (logged it: `descMinWidth` climbing 252‚Üí380 mid-drag) yet cells render the stale width; `getColumnWidths` would compute it correctly ‚Äî it just isn't re-run. The stale memo lives **inside** the library (its `useColumns` keys on the `columns` array identity), so `"use no memo"` on our component does nothing. ~40 min lost: hardcoding the same sizing at mount worked (proving the algorithm), only a runtime change failed ‚Äî isolating "mount vs post-mount" was the tell.
- **Rule**: To apply a column-width change at runtime, **remount the grid** with a `key` tied to the width signature (`key={JSON.stringify(widths)}`) ‚Äî nothing short of a remount reliably re-lays-out columns. Because a remount resets scroll/focus and re-runs virtualization, make resizing **commit-on-release** (persist + remount on pointer-up), not live-per-move, and give in-drag feedback with a cheap overlay (a `fixed` vertical guide line at the cursor) instead of resizing the grid live. Anchor in-header affordances to `.dsg-cell-header` (the library gives it `position:absolute`) by leaving the title wrapper **non-positioned** ‚Äî a `position:relative` wrapper re-anchors `absolute` children to the shrink-wrapped title content, not the cell edge. Live drag-resize without a remount is not achievable in this library without a fork.
- **Applies to**: implement, impl-review, verify, frontend grid/table work
