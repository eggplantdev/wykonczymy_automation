# Kosztorys client share view — design

**Change ID:** `kosztorys-client-share`
**Slice:** S-13 (band 2 — import/export)
**Linear:** EX-532
**Date:** 2026-07-18
**Status:** active — resumed 2026-07-20 (owner: domain rework "mostly settled"; the editor
stays under development and that is accepted). The original wait was to avoid the client view
becoming a second thing to keep in sync — resolved by the single-source rule below.
Two corrections applied on resume, marked inline: the token's home (§1) and the
"disconnected from financials" premise (Problem).

## Problem

The owner wants to hand a client a **live, read-only view** of a kosztorys — not a
frozen offer, but a URL the client can reopen to see current progress (per-etap
execution) over the life of the job. The hard constraint: it must **never** leak
the **subcontractor cost view** — the z-narzędziami / bez-narzędzi prices.

Note: kosztorys v2 has **no marża / internal-cost field**. (Corrected 2026-07-20: the
original "v2 is disconnected from the app's financials" no longer holds — EX-541 added a
**read-only** recon bridge comparing the kosztorys client-view net against the investment's
robocizna/rabat transaction sums, `lib/kosztorys/reconciliation.ts:66`. It is a comparison
surface, never a data feed: `lib/db/calculate-margin.ts:13` still derives margin purely from
transactions. The client payload carries no reconciliation at all — see plan Phase 2.)
The only sensitive data in the model is the subcontractor
cost view: `CostVariantT` (`w_tools` / `own_tools`), the per-investment
`wToolsCoeff` / `ownToolsCoeff`, and the per-item `wToolsOverride*` /
`ownToolsOverride*`. The client-safe figure is `clientPrice`.

Explicitly rejected: reusing the owner's editor with column RBAC. A working-UI
toggle is transient and error-prone as a security boundary; one stray toggle on a
live public URL leaks margin. The owner asked for a _separate_ view instead.

## Decisions

- **One shared data source, a distinct client render** — same live rows as the
  editor, projected to a client-safe shape. Not the editor + permissions.
- **The no-leak guarantee is structural, not a runtime filter** (core principle).
- **Live, not a snapshot** — reads current rows each visit; reuses the existing
  kosztorys cache tag so an editor edit revalidates the public page.
- **Per-etap progress is visible** — the stage-by-stage execution is the whole
  point; the client sees how the work is going.
- **Scope = this view only.** Google Sheet export and PDF export are separate
  later slices. PDF may be dropped entirely — a live link beats a static file.

## Architecture

### 1. The token — its own `kosztorys-shares` table

**Corrected 2026-07-20 (owner approved).** The original "add `shareToken` to `kosztoryses`" was
wrong: `kosztoryses` (`collections/sheets.ts:13`) is the **v1 Google-Sheet link row**, where
`googleSheetId` is required+unique and `investment` is nullable — so a v2 kosztorys with no linked
sheet would have no row to carry a token and could never be shared. Kosztorys v2 has no top-level
collection; its data keys on `investment`. The token therefore lives in a new `kosztorys-shares`
table keyed on `investment`, where revoke is a row delete. See plan Phase 1.

- Generated by OWNER/ADMIN via a "Udostępnij klientowi" action.
- Value: unguessable random, `randomBytes(24).toString('base64url')`.
- **Revocable — two ways, both OWNER/ADMIN:**
  - **Revoke (turn sharing off):** delete the share row. The link dies immediately;
    no new one is issued. This is the explicit kill switch.
  - **Rotate (re-share, invalidate old):** mint a new token over the existing row →
    the old link dies, a fresh link takes over.
- No share row → no public access, full stop (the public read returns 404).
- Only new stored state this change introduces.
- **The owner's preview does not depend on this row** (owner, 2026-07-20) — „Podgląd
  dla klienta" is permanently available from the kosztorys, shared or not. See plan Phase 3.

### 2. Structural no-leak — pin the price view to `'client'`

Prices are **computed live**, not stored — `calc.ts` persists only the inputs
(`clientPrice`, coeffs, overrides) and derives every money figure through pure
functions that take a `view` argument:

```ts
type PriceViewT = 'client' | 'w_tools' | 'own_tools'
viewPrice(row, view) // view==='client' → clientPrice; else subcontractorPrice(row, view)
netForQtyForView(row, qty, view)
```

So the **entire leak surface is that single `view` argument.** The client path
reuses the _same_ `calc.ts` functions but **hardcodes `view: 'client'`** — a
constant, never derived from the owner's toggle, never passed in from outside.
With `view` pinned to `'client'`, the `w_tools` / `own_tools` branches are never
reached: the subcontractor price is not "filtered out", it is **never computed**.
This is a stronger guarantee than RBAC — the one knob that could produce a
subcontractor price is a literal constant in the client path.

Reinforce it at the type layer: a new `ClientKosztorysViewT` DTO carries **only**
client-safe fields (no `CostVariantT`, no coeffs, no override values), and the
public page imports only that type — so even the inputs to a subcontractor price
never enter the client render's module graph.

**Client-safe field set** (sheet vocabulary): section + item description,
Przedmiar, j.m., Cena j.m. (the client price — no tool-mode split), Wartość
netto przedmiar, per-etap progress (`D:M`), Pomiar z natury, Wartość netto pomiar
z natury; footer block: wartość netto, suma transzy, Pozostałe koszty, the two
Materiały rows, aktualnie do zapłaty R + M (netto/brutto), R … suma prac
wykonanych; plus the section pie chart.

### 3. Public read path

`getClientKosztorysView(token)` — an **unauthenticated** server function (no
`requireAuth`, unlike `getKosztorysTree`). Looks up the kosztorys by `shareToken`,
resolves its `investment`, builds the tree, projects via `toClientView`. Absent
or unknown token → 404.

### 4. Public route (outside auth)

New no-auth route group `(share)` with `/k/[token]/page.tsx` and its own bare
layout (no app chrome, no nav). `robots` → `noindex`. Renders the DTO with a
dedicated read-only presentational component `ClientKosztorysTable` (+ footer +
pie) — pure display, no editor imports, no mutations.

## Also in scope

- **Preview.** The owner can preview exactly what the client will see — an authenticated
  "Podgląd dla klienta", reachable by a **permanent direct link from the kosztorys,
  independent of share state** (owner, 2026-07-20). Near-free given the single-source
  design: it mounts the _same_ client projection (read-only grid, `clientVisible`
  columns, `view: 'client'`) the public route serves, just behind auth. No second
  render. This is the owner's confidence check that nothing leaks.

## Future (nice-to-have, not this slice)

## Out of scope

- Google Sheet export (separate slice — the owner flagged it as the hard one).
- PDF export (separate slice; may be dropped).
- Any per-kosztorys configuration of the visible field set — the client view is a
  single fixed, client-safe render for every kosztorys.

## Reuse & sync strategy (revised 2026-07-18)

**Driving constraint (owner):** the editor is under active development — do **not**
create a second render to keep in sync. **One source of truth, one read-only
projection.** Not "one component vs two" — "how many sources of truth?"

**Single-source (nothing to keep in sync):**

- **Derivation** — `calc.ts`, already shared; both views call the same functions.
- **Column definitions + cell formatters** — the _same_ `column-config.ts` /
  `kosztorys-v2-columns.tsx` and money/unit formatters. Add a column to the editor
  → it exists once → the client view picks it up automatically.

**Client visibility — declarative, not imperative:**

- Each column declares `clientVisible: boolean` **in the shared column config**.
  The client view filters to `clientVisible` columns. Adding a column forces a
  one-line, conscious "client-safe?" decision in the _same place_ it's defined —
  not a second file.
- ❌ Never a runtime `{mode !== 'client' && <SubcontractorPrice/>}` in render code:
  scattered, easy to miss, and the dangerous data is still in props.
- ✅ Defense-in-depth: `view` pinned to `'client'` means the subcontractor price is
  **never computed**, so a mistaken flag still can't leak a number that doesn't exist.

**Component reuse (owner OK, 2026-07-18):** reuse the editor grid in a **read-only
mode** — it's cheap when read-only (none of the autosave / optimistic-store / undo
machinery is active). So the client route can mount the same grid with editing off,
fed `clientVisible` columns + `view: 'client'` data, rather than a separate table.
Plan confirms this at build time once the editor has settled.
